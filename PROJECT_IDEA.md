# PROJECT_IDEA — «Кадр»: событийная камера для общих фото

> **Аналог pov.camera для России.** Гость заходит по QR/ссылке без установки приложения,
> снимает на «плёночную» камеру, все кадры летят в общую галерею события. Хост видит
> событие глазами всех гостей.
>
> Документ написан как **ТЗ для AI-агентов** (Claude Code + GSD-субагенты), а не как питч.
> Это Layer 1 методологии Spec-First. Следующий слой — `SPECIFICATION.md`.

> [!warning] Главное архитектурное ограничение (читать первым)
> Проект — **реальный коммерческий запуск в РФ**, поэтому подчиняется 152-ФЗ.
> **Supabase Cloud и Vercel использовать НЕЛЬЗЯ** для персональных данных граждан РФ:
> их серверы за рубежом, а ст. 18 ч. 5 152-ФЗ требует первичной записи и хранения ПДн
> в базах **на территории России**. Решение: **self-hosted Supabase + деплой на российском
> облаке** (Yandex Cloud / VK Cloud / Selectel). Весь стек и методология сохраняются.
> Раздел 11 — отдельная карта соответствия. Я не юрист: финальные юр-документы и оценку
> уровня защищённости должен подтвердить профильный специалист по ПДн.

## 01 · Проблема

На событиях (свадьбы, дни рождения, корпоративы, выпускные) фото оказываются разбросаны:

- Профессиональный фотограф отдаёт снимки через **2–6 недель** — момент остыл.
- Гостевые кадры тонут в десятках разных телефонов и не собираются в одно место.
- Сборка через общий чат / облачную папку = хаос: дубли, разные ракурсы вперемешку,
  никто не знает, кто что снял.
- Существующие решения (pov.camera, Joinmymoment, Shared) — **только на английском**,
  оплата зарубежными картами, серверы за рубежом → для российского рынка фактически
  недоступны и **несовместимы со 152-ФЗ**.

**Цена боли:** организатор события платит фотографу 30–80 тыс. ₽, ждёт недели и всё равно
теряет «живые» гостевые кадры — самое ценное, что есть только у участников.

---

## 02 · Решение

Веб-первый вход для гостей + нативная iOS-камера для атмосферы «одноразовой плёнки».

Пошаговый сценарий (это основа модулей спецификации):

1. **Хост создаёт событие** в iOS-приложении: название, дата, обложка, стиль камеры,
   лимит кадров на гостя, время «проявки».
2. Система генерирует **уникальный QR-код и короткую ссылку** события.
3. **Гость сканирует QR** → открывается web/PWA-камера в браузере **без установки**
   (аналог App Clip). Даёт согласие на обработку → снимает.
4. Каждый кадр **автоматически загружается** в общую галерею события в реальном времени.
5. До момента **«проявки»** фото скрыты (саспенс одноразовой плёнки); в назначенное время
   галерея раскрывается всем.
6. Хост и гости **фильтруют галерею по автору** («POV конкретного гостя»), отмечают
   избранное, скачивают.
7. Апселл: **фотокнига** по итогам события, доп-лимиты, видео-клипы (v2).

**Ключевые отличия для РФ:** русский интерфейс, оплата ЮKassa/СБП, серверы и фото в РФ,
полное соответствие 152-ФЗ «из коробки».

## 04 · Целевая аудитория

**Основной сегмент — организаторы событий (хосты):**

- **Кто:** невесты/женихи, организаторы свадеб, ивент-менеджеры, родители (детские
  праздники), HR/корпоративы. 25–40 лет, iPhone, платёжеспособны.
- **Задача:** собрать все гостевые фото в одном месте сразу, без недель ожидания.
- **Текущие инструменты:** профессиональный фотограф + общий чат/Google-папка (костыль).

**Вторичный сегмент — гости события:**

- **Кто:** участники 18–60, разные устройства (поэтому вход — web/PWA, не только iOS).
- **Задача:** быстро снять и поделиться, **не устанавливая приложение**.
- **Барьер, который снимаем:** «ещё одно приложение ставить не хочу» → вход по ссылке.

**Третичный (v2):** event-агентства и площадки (white-label, брендирование галереи).

## 05 · Архитектура

### Слои системы

```
┌───────────────────────────────────────────────────────────────┐
│ КЛИЕНТЫ                                                          │
│  • iOS-приложение (SwiftUI)  — хост: создание событий, камера,   │
│                                 «плёночные» фильтры, просмотр     │
│  • Web/PWA (Next.js)         — гость: вход по QR/ссылке,          │
│                                 камера в браузере, без установки  │
└───────────────────┬───────────────────────────────────────────┘
                     │ HTTPS (TLS), JWT
┌───────────────────▼───────────────────────────────────────────┐
│ BACKEND — self-hosted Supabase (в российском облаке)            │
│  • Auth        — телефон (OTP, рос. SMS-провайдер) / email       │
│  • Postgres    — события, гости, фото-метаданные, согласия       │
│  • Storage     — S3-совместимое хранилище фото (рос. объектное)  │
│  • Realtime    — live-обновление галереи                         │
│  • Edge Funcs  — QR-генерация, вебхуки оплаты, «проявка»          │
└───────────────────┬───────────────────────────────────────────┘
                     │
┌───────────────────▼───────────────────────────────────────────┐
│ ВНЕШНИЕ СЕРВИСЫ (все — российские, кроме APNs)                   │
│  • ЮKassa / СБП — приём оплаты в ₽                               │
│  • SMS Aero / SMSC — OTP-коды                                    │
│  • Yandex AppMetrica / Метрика — аналитика (НЕ Google/Firebase)  │
│  • APNs (Apple) — пуши на iOS (см. раздел 11: трансграничка)     │
└─────────────────────────────────────────────────────────────────┘
```

### Стек технологий с обоснованием

| Слой | Технология | Почему |
|---|---|---|
| iOS | **Swift / SwiftUI** | нативная камера, плёночные фильтры (Core Image), производительность, App Clip-подобный UX |
| Гостевой клиент | **Next.js (PWA)** | вход без установки, камера через `getUserMedia`, совпадает с методологией курса |
| Backend | **self-hosted Supabase** | Postgres + Auth + Storage + Realtime в одном; **разворачивается в РФ** → 152-ФЗ |
| Хостинг | **Yandex Cloud / VK Cloud / Selectel** | данные физически в РФ (локализация ст. 18.5) |
| Хранилище фото | **Yandex Object Storage / VK Cloud S3** | S3-совместимо с Supabase Storage, в РФ |
| Оплата | **ЮKassa + СБП** | работает с рос. картами и СБП; Stripe недоступен |
| Пуши iOS | **APNs** | единственный путь для iOS (нюанс трансграничной передачи — раздел 11) |
| Аналитика | **AppMetrica / Яндекс.Метрика** | российская, без вывоза данных за рубеж |
| IaC/CI | self-hosted CI (GitLab/Gitea) или GitHub Actions с деплоем в рос. облако | избегаем Vercel |

> [!note] Отклонение от методологии курса
> Курс по умолчанию = Next.js + **Supabase Cloud + Vercel**. Для РФ-запуска с ПДн это
> заменяется на **self-hosted Supabase + российское облако**. Сам Next.js, паттерны,
> GSD-субагенты и Spec-First — без изменений.

### Модули (вход → выход)

| Модуль | Вход | Выход |
|---|---|---|
| **Auth** | телефон/email хоста | сессия (JWT), запись `users` |
| **Events** | параметры события | `event` + QR + ссылка |
| **Guest Join** | QR/ссылка + согласие | гостевая сессия (анонимная, токен) |
| **Capture** | кадр с камеры | загруженное фото в Storage + метаданные |
| **Reveal** | таймер «проявки» | переключение видимости галереи |
| **Gallery** | event_id | список фото, фильтр по автору, избранное |
| **Billing** | выбор тарифа | оплата ЮKassa → апгрейд лимитов |
| **Consent & Privacy** | действия пользователя | журнал согласий, обработка запросов субъектов |
| **Retention** | дата окончания + тариф | автоудаление события по сроку |

## 09 · Риски

| Риск | Вероятность | Митигация |
|---|---|---|
| **Фото лиц = биометрия (ст. 11)** | средняя | в MVP **нет распознавания лиц**; отдельное согласие на обработку изображений; зафиксировать запрет на face-recognition в архитектуре |
| **Третьи лица на фото без согласия** | высокая | согласие гостя при входе + обязательство хоста уведомить участников + быстрый механизм удаления по запросу |
| **Локализация: облако вне РФ** | высокая → критично | self-hosted Supabase в Yandex/VK Cloud; запрет деплоя в Vercel/Supabase Cloud в правилах проекта |
| **Трансграничка через APNs/CDN/аналитику** | средняя | аналитика только российская; уведомление РКН о трансграничной передаче для APNs; CDN российский |
| **Несовершеннолетние на фото** | средняя | возрастная отметка события; согласие родителей; настройка приватной (не публичной) галереи по умолчанию |
| **Утечка фото (репутация + штраф)** | средняя | шифрование at-rest, подписанные URL с TTL, RLS на каждую таблицу, аудит Security-агентом |
| **Юнит-экономика хранения** | средняя | авто-retention по тарифу, ленивая генерация превью, lifecycle-политики в Object Storage |
| **iOS-only сужает охват гостей** | низкая | гости и так заходят через web → охват не страдает; нативка только у хоста |

## 11 · Соответствие 152-ФЗ (карта соответствия) ⚠️

> Это центральный раздел проекта. Каждый пункт — требование, которое закладывается в
> архитектуру **до** написания кода. Я не юрист — пункты 1, 2, 9 обязательно
> согласовать с профильным специалистом по защите ПДн.

### 11.1 Локализация (ст. 18 ч. 5) — критично

- Первичная запись, хранение и обработка ПДн граждан РФ — **только в БД на территории РФ**.
- **Запрет:** Supabase Cloud, Vercel, Firebase, любой зарубежный хостинг для ПДн.
- **Решение:** self-hosted Supabase + Yandex Cloud / VK Cloud / Selectel; фото — в
  российском S3-совместимом Object Storage.
- Закрепить запрет деплоя за рубеж в `.claude/rules/` и в CI.

### 11.2 Биометрия (ст. 11) — серая зона, обходим сознательно

- Фото лица = **биометрические ПДн, если используются для установления личности**.
- В MVP **НЕТ распознавания/тегирования лиц** → остаёмся вне ст. 11 и ЕБС (572-ФЗ).
- Если в v2 появится face-recognition — отдельный строгий режим: письменное согласие,
  возможная интеграция с ЕБС. **До этого — запрет в архитектуре.**

### 11.3 Согласие на обработку ПДн

- **Хост:** согласие при регистрации (имя, телефon/email, цель — оказание услуги).
- **Гость:** согласие при входе по QR **до** первого кадра (загрузка фото, где могут быть
  изображены третьи лица).
- Согласие — информированное, с версионированием текста; всё пишется в `consents`
  (что, когда, версия, IP, UA).
- Отзыв согласия → удаление данных субъекта.

### 11.4 Третьи лица на фото

- Гость подтверждает, что вправе фотографировать участников; хост обязуется уведомить
  гостей об использовании сервиса.
- Галерея по умолчанию **приватная** (не публичная ссылка в индекс).
- Быстрый механизм «удалить фото со мной» по запросу.

### 11.5 Уведомление Роскомнадзора

- Подать **уведомление об обработке ПДн в РКН до начала обработки** (фаза 0).
- Указать цели, категории субъектов, перечень действий, **трансграничную передачу**
  (см. 11.7), меры защиты.

### 11.6 Политика и документы (опубликовать до запуска)

- Политика обработки ПДн (публичная страница).
- Согласия (формы) для хоста и гостя.
- Положение о мерах защиты ПДн (внутренний документ оператора).

### 11.7 Трансграничная передача (ст. 12)

- **APNs (Apple):** device-токены уходят за рубеж — отметить в уведомлении РКН как
  трансграничную передачу; минимизировать состав данных (токен ≠ имя/фото).
- **Аналитика:** только российская (AppMetrica/Метрика), НЕ Google/Firebase.
- **CDN:** российский; не отдавать фото через зарубежный CDN.

### 11.8 Уровень защищённости (ПП РФ № 1119 + Приказ ФСТЭК № 21)

- Определить **уровень защищённости (УЗ)** ИСПДн. Без биометрии и при небольшом объёме —
  вероятно УЗ-3/УЗ-4 (подтвердить у специалиста).
- Меры: шифрование at-rest и in-transit (TLS), подписанные URL с TTL для фото,
  RLS на всех таблицах, журналирование доступа, разграничение ролей, резервные копии в РФ.

### 11.9 Права субъектов и сроки хранения (ст. 5, 14)

- Хранить **не дольше, чем нужно**: авто-retention события по тарифу (`expires_at`) →
  автоудаление фото и метаданных.
- Реализовать запросы субъекта: доступ, уточнение, удаление, отзыв согласия
  (`deletion_requests`), срок ответа — по закону.
