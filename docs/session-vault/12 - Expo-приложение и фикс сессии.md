# 12 — Expo-приложение хоста и фикс сессии гостя

[[00 - Карта проекта|← к карте]]

## Зачем Expo (а не SwiftUI)

Есть два приложения хоста:
- `ios/` — нативное SwiftUI ([[10 - iOS-приложение хоста]]), требует Xcode-подписи.
- `mobile/` — **Expo/React Native** — запуск через **Expo Go** на iPhone **без Xcode, без подписи, без сертификатов**. Это то, что реально запустили на телефоне пользователя.

## Mobile (`mobile/`) — Expo SDK 54

> [!warning] SDK 54, не 56
> create-expo-app дал SDK 56 → Expo Go из App Store ругался «Project is incompatible». Понизили до **SDK 54** (react-native 0.81.5) — совместим с публичным Expo Go. `expo install --fix` подтянул версии.

**Запуск:**
```bash
cd mobile
export NVM_DIR="$HOME/.nvm"; . "$NVM_DIR/nvm.sh"; nvm use 22.22.2
npx expo start          # QR в терминале → Expo Go на iPhone
```
Конфиг (IP Mac, ключ) — `mobile/src/config.ts`. Бэкенд должен быть поднят.

**Экраны:** логин (+согласие), мои события, создание события, QR-код (react-native-qrcode-svg), галерея (проявка, избранное). Демо-вход = анонимная сессия в роли хоста.

## Баг: «не удалось создать безопасную сессию» (2-й гость)

**Симптом:** первый гость заходит ок, второй с другого устройства — ошибка сессии.

**Корень:** гостевой веб ходил к Supabase по `https://192.168.50.124:8444` (самоподписанный caddy). У второго устройства **нет доверенного CA caddy** → браузер блокирует HTTPS → `signInAnonymously` падает. Доказано: с `ignoreHTTPSErrors` (= доверенный CA) всё работает (signup 200 → join 201).

**Фикс:**
1. `joinEvent.ts` — анонимный вход через **прямой fetch** `/auth/v1/signup` (не supabase-js signInAnonymously) — устойчивее, не зависит от self-signed TLS внутри клиента. Понятные ошибки rate_limited/network.
2. `config.toml` — `anonymous_users` 30→200/час (гости одного события за одним Wi-Fi = один IP).
3. **Локальный режим → HTTP-IP** (`http://192.168.50.124:3000` + `:54321`). Гости вступают **без сертификата** с любого устройства. Проверено: 3 независимых гостя (signup 200 → join 201).
4. `camera` — на не-HTTPS показывает понятное сообщение (камера требует HTTPS; на проде домен kadr.ru), вступление и галерея при этом работают.

## Компромисс HTTP vs HTTPS (локально)

| Режим | Вступление с любого устройства | Камера на телефоне |
|---|---|---|
| HTTP-IP (сейчас) | ✅ без сертификата | ❌ нужен HTTPS |
| caddy HTTPS + CA на устройстве | ✅ | ✅ |
| публичный туннель | ✅ | ✅ (заблокирован политикой безопасности) |

На **проде** вопрос снимается — реальный HTTPS-домен (РФ-облако): и вступление, и камера работают для всех без плясок.
